# Project Brief: West Africa Daily Briefs — Webapp

## Overview

Build a public-facing web application that serves as the browsing and reading interface for the West Africa Daily Briefs research reports. The site pulls data from the `west-africa-daily-briefs` GitHub repo and presents it in a clean, branded, commercially polished experience.

**Live data source:** `https://github.com/Oswald-Benjamin/west-africa-daily-briefs`
**JSON feeds:** `https://raw.githubusercontent.com/Oswald-Benjamin/west-africa-daily-briefs/master/site/data/`

- **Cards:** Rounded corners (8-12px), subtle border, dark card background (`#223239`). Hover state lifts with shadow.
- **Buttons:** Gold background with dark text, or ghost/outline style with gold border.
- **Navigation:** Minimal top nav with Webara Studio-style logo treatment.
- **Spacing:** Generous padding. This is a premium feel — don't cramp content.
- **Icons:** Use emoji for topic categories (📱₿🌾⚡📈🌍📦) — they're already in the data.

## Pages & Features

### 1. Home (`/`)

**Hero section:**
- Headline: "West Africa Daily Briefs"
- Subtitle: "Daily research on West African business, markets, and investment opportunities"
- Gold CTA button: "Read Today's Brief"
- Secondary link: "Browse Archive"

**Today's Brief card:**
- Fetches the most recent brief from `latest.json`
- Shows: date, topic emoji, title, sentiment indicator, 3 key takeaways
- Click through to full brief

**Week at a Glance:**
- Grid of the last 7 days' briefs (emoji + title + one-line summary)
- Each card links to the full brief

**Topic Categories:**
- 7 category cards (Tech, Crypto, Agribusiness, Energy, Stocks, Governance, Trade)
- Each shows the count of briefs and the most recent brief title
- Click through to filtered archive

**Newsletter signup:**
- Simple email input + gold CTA button
- Text: "Get the weekly digest. Every Sunday."

### 2. Brief Reader (`/briefs/[slug]`)

- Full rendered Markdown content of a single brief
- YAML frontmatter displayed as metadata: date, topic, category, sentiment, sources
- "Previous Brief" / "Next Brief" navigation at bottom
- Sidebar or top bar: category tag, sentiment badge, share buttons
- Clean typography optimised for reading (max-width ~720px, good line height)

### 3. Archive (`/archive`)

- Filterable list of all briefs
- Filters: category (topic), month, sentiment
- Sort: newest first (default), oldest first
- Pagination or infinite scroll
- Each row: date, emoji, title, category tag, sentiment dot

### 4. Category Pages (`/category/[slug]`)

- Filtered view for each of the 7 topics
- Same layout as archive but pre-filtered
- Shows total count and date range

### 5. Newsletter (`/newsletter`)

- Archive of weekly newsletters
- Each newsletter rendered as a full page
- Signup form prominently placed

### 6. About (`/about`)

- What this project is
- The 7 topics and schedule
- Link to GitHub repo
- Link to Webara Studio

## Data Architecture

The site reads from three JSON files generated by the `build_feeds.py` script:

### `briefs.json`
Array of all briefs with frontmatter fields:
```json
{
  "date": "2026-06-05",
  "day": "Friday",
  "topic": "IPOs & Stock Market Watch",
  "category": "stocks",
  "week_number": 23,
  "sentiment": "mixed",
  "key_takeaways": ["...", "...", "..."],
  "opportunities": ["...", "..."],
  "watch_list": ["...", "..."],
  "sources": ["https://..."],
  "_file": "briefs/2026/06/2026-06-05-stocks.md"
}
```

### `latest.json`
Object keyed by category, each value is the most recent brief in that category.

### `categories.json`
Object keyed by category, each value is an array of all briefs in that category.

### Fetching Strategy
- **Option A (recommended):** Fetch JSON at build time via `getStaticProps` or server components. Rebuild via GitHub webhook when new briefs are pushed.
- **Option B:** ISR (Incremental Static Regeneration) with a 1-hour revalidation interval.
- **Option C:** Client-side fetch on page load (simplest, but worse SEO).

**Recommended: Option B (ISR)** — good balance of freshness and performance.

## Markdown Rendering

Briefs are stored as Markdown files. The site needs to:

1. Fetch the raw Markdown from GitHub (`https://raw.githubusercontent.com/...`)
2. Render it as HTML using a Markdown library (`react-markdown` or `marked`)
3. Style the rendered HTML to match the brand (headings in gold, links in gold, blockquotes styled, etc.)
4. Support the YAML frontmatter being stripped before rendering

## File Naming Convention
